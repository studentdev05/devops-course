## 🎯 Мета роботи

Навчитись орієнтуватись у мережевих налаштуваннях віртуальної машини: переглядати IP-адреси та відкриті порти, розуміти критичну різницю між прив'язкою сервісу до `127.0.0.1` та `0.0.0.0`, перевіряти доступність сервісу за допомогою `curl`.

> **Контекст:** Ми підготуємо мережу так, щоб у майбутньому отримати доступ до інтерфейсу нашого AI-агента OpenClaw (порт 18789), зберігши його захищеним від зовнішнього світу.

## 🛠 Покрокова інструкція

### Крок 1: Запускаємо VM та перевіряємо мережу

Підключіться до вашої віртуальної машини, як ви це робили у Темі 2:

```bash
# На хості:
cd ~/devops-ai-assistant/
vagrant up
vagrant ssh
```

Тепер, знаходячись всередині VM, перевіримо її мережеві інтерфейси:

```bash
# Детальний вивід всіх адаптерів та їхніх IP
ip addr

# Швидкий перегляд тільки IP-адрес
hostname -I
```

**Очікуваний результат:** У виводі `ip addr` ви маєте знайти як мінімум:

- Інтерфейс `lo` (loopback) з адресою `127.0.0.1`.
- Інтерфейс `enp0s3` з адресою `10.0.2.15` — NAT-адаптер VirtualBox для виходу в інтернет.
- Інтерфейс `enp0s8` з адресою `192.168.56.10` — IP-адреса нашого сервера.

### Крок 2: Хто слухає порти просто зараз?

Перш ніж запускати свої сервіси, подивимось що вже "відкрито" на чистій Ubuntu:

```bash
sudo ss -tlnp
```

**Очікуваний результат:**

```text
State    Recv-Q  Send-Q  Local Address:Port   Peer Address:Port  Process
LISTEN   0       128           0.0.0.0:22          0.0.0.0:*     users:(("sshd",pid=698,fd=3))
LISTEN   0       4096    127.0.0.53%lo:53          0.0.0.0:*     users:(("systemd-resolve",pid=586,fd=14))
LISTEN   0       128              [::]:22              [::]:*     users:(("sshd",pid=698,fd=4))
```

Три рядки — це нормально для свіжої Ubuntu:

- `sshd` на `0.0.0.0:22` — SSH-сервер, наш єдиний "вхід" до VM.
- `systemd-resolve` на `127.0.0.53:53` — вбудований DNS-резолвер, живе тільки на loopback.
- `sshd` на `[::]:22` — той самий SSH, але для IPv6.

OpenClaw поки що не запущений — порту `18789` немає. Запам'ятайте цю картину, вона нам знадобиться для порівняння.

### Крок 3: Симуляція агента — прив'язка до 0.0.0.0

Щоб зрозуміти роботу мережі, ми не будемо розгортати повного OpenClaw агента. Ми використаємо Python для створення короткоживучого "фейкового" веб-сервера.

```bash
# Запускаємо веб-сервер на порту 18789, прив'язаний до всіх інтерфейсів
# Символ & запускає процес у фоновому режимі
python3 -m http.server 18789 --bind 0.0.0.0 &
```

Символ `&` запускає процес у фоні — термінал одразу повертається. Але фоновий процес продовжує писати свої повідомлення в той самий екран, тому може виглядати так:

```text
[1] 3893
vagrant@devops-sandbox:~$ Serving HTTP on 0.0.0.0 port 18789 ...
```

- `[1]` — номер завдання (job) у цій сесії bash. Саме його ми використаємо пізніше у `kill %1`.
- `3893` — PID (Process ID), унікальний номер процесу в системі.

Це нормально. Просто натисніть **Enter** — отримаєте чистий рядок і продовжуйте в тому самому терміналі:

Перевірте що порт з'явився:

```bash
sudo ss -tlnp | grep 18789
```

**Очікуваний результат:**

```text
LISTEN 0  5  0.0.0.0:18789  0.0.0.0:*  users:(("python3",pid=3932,fd=3))
```

Колонка `0.0.0.0:*` праворуч — це **Peer Address:Port**, тобто "від кого приймати з'єднання". `0.0.0.0:*` означає "від будь-якої IP, з будь-якого порту". Це нормальне значення для сокета в стані LISTEN — він ще ні з ким не з'єднаний.

Перевірте доступ зсередини VM:

```bash
curl -I localhost:18789
# Очікуваний результат: HTTP/1.0 200 OK
```

Тепер відкрийте браузер **на вашому ноутбуці (хості)** і перейдіть за адресою:

```text
http://192.168.56.10:18789
```

**Очікуваний результат:** Сторінка відкривається — ви бачите лістинг директорії.

> ⚠️ **Проблема безпеки:** Сервіс доступний будь-кому, хто може дотягнутися до VM. Для тестового сервера це прийнятно, але для AI-агента — ні. Переходимо до правильного налаштування.

### Крок 4: Правильна ізоляція агента — прив'язка до 127.0.0.1

Зупинимо відкритий сервер:

```bash
kill %1
```

> 💡 **Чому `%1`?** Коли ви запустили процес з `&`, bash призначив йому номер завдання `[1]`. Команда `kill %1` каже bash: "зупини завдання номер 1 у цій сесії". Подивитись усі активні фонові завдання можна командою `jobs`.

Тепер запускаємо **правильно** — прив'язуємо тільки до loopback:

```bash
python3 -m http.server 18789 --bind 127.0.0.1 &
```

Перевірте `ss` — адреса має змінитись:

```bash
sudo ss -tlnp | grep 18789
# Очікуваний результат: LISTEN ... 127.0.0.1:18789 ... python3
```

Перевірте доступ зсередини VM — має працювати:

```bash
curl -I localhost:18789
# Очікуваний результат: HTTP/1.0 200 OK
```

Тепер спробуйте зайти з браузера на ноутбуці на `http://192.168.56.10:18789`.

**Очікуваний результат:** Сторінка не відкривається. Сервер є, він працює — але тільки для програм всередині самої VM.

> 💡 Саме так налаштований реальний OpenClaw на продакшн-сервері. У наступній **Темі 4** ми навчимось підключатись до нього через SSH-тунель — зашифровано і безпечно.

Зупиніть фоновий процес:

```bash
kill %1
```

### Крок 5: Перевірка доступності порту через curl

Потренуємось використовувати `curl` для діагностики — це навичка, яка знадобиться щоразу, коли щось "не працює".

```bash
# Перевірка порту, на якому нічого не запущено
curl -I localhost:18789
```

**Очікуваний результат:**

```text
curl: (7) Failed to connect to localhost port 18789: Connection refused
```

`Connection refused` — не помилка вашої VM. Це точна і корисна інформація: порт існує, але ніхто його не слухає. Запустіть сервер знову і переконайтесь, що відповідь зміниться:

```bash
python3 -m http.server 18789 --bind 127.0.0.1 &
curl -I localhost:18789
# Очікуваний результат: HTTP/1.0 200 OK

kill %1
```

### Крок 6: UFW — коротка демонстрація

Подивимось на стан фаєрволу на нашій VM:

```bash
sudo ufw status
# Очікуваний результат: Status: inactive
```

UFW вимкнений — і це нормально для локальної VM. Як ми розібрали в теорії, наш захист забезпечується прив'язкою до `127.0.0.1`, а не фаєрволом.

Для розуміння того, як виглядає UFW в реальній роботі, увімкнемо і одразу вимкнемо його:

```bash
# ЗАВЖДИ спочатку дозволяйте SSH — інакше втратите доступ до сервера!
sudo ufw allow 22/tcp

# Увімкнути — підтвердіть 'y'
sudo ufw enable

# Перевірити стан
sudo ufw status verbose

# Вимкнути — щоб не заважав майбутнім лабораторним
sudo ufw disable
```

**Зверніть увагу:** `sudo ufw status verbose` показує дефолтну політику (`deny incoming`) і список відкритих портів. Саме так виглядає налаштований фаєрвол на реальному VPS.
