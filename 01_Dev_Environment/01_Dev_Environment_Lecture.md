## 🎯 Бізнес-задача

Ми готуємось розгортати AI-асистента. Але є проблема: нам потрібно встановлювати пакети, змінювати конфігурації, перезапускати сервіси — і якийсь із цих кроків обов'язково піде не так. Якщо ми будемо робити це на реальному сервері (Production) — один невдалий крок, і агент перестане відповідати всім реальним користувачам.

**Нам потрібна «пісочниця»** — ізольоване середовище-копія реального сервера, яке можна зламати без наслідків, а потім відновити за хвилину і спробувати знову.

**Зв'язок з проєктом:** У `My_AI_Assistant` є Ansible playbook та Terraform конфіги — вони налаштовують реальний VPS. Щоб навчитися їх використовувати без ризику, нам потрібна локальна VM, яка максимально нагадує Production-сервер. Саме цим ми займемось сьогодні.

---

## 🎭 Метафора заняття: Тренувальний полігон vs реальна дорога

Перш ніж молодий водій сідає за кермо на жвавій трасі, він тренується на закритому полігоні. Можна зробити будь-яку помилку — перевернутися в кювет, заїхати на бордюр — і нічого серйозного не станеться. Після кожної аварії можна «скинути» полігон у початковий стан і спробувати знову.

**Dev Environment — це такий самий полігон для DevOps-інженера.** Production — жвава траса, де їздять реальні люди.

> 💡 **Головне правило DevOps:** *Ніколи не тестуй на Production.* Для цього існує Dev Environment.

## 🧠 Необхідні знання

### 1. Середовища розробки: Dev, Staging, Production

У будь-якому серйозному проєкті є кілька окремих ізольованих середовищ — **environments**. Кожне має свою роль, своє залізо (або VM) та свій рівень доступу.

| Середовище     | Хто використовує           | Ризик  | Дані              | Мета                            |
| -------------- | -------------------------- | ------ | ----------------- | ------------------------------- |
| **Development**  | Тільки розробник / студент | Низький | Тестові / фейкові | «Пісочниця» — можна ламати      |
| **Staging**      | QA, тімлід, замовник       | Середній| Копія реальних   | «Генеральна репетиція» перед релізом |
| **Production**   | Реальні користувачі        | Високий | Справжні дані     | Бойова система — ламати не можна |

Сьогодні ми будуємо **Development** середовище — локальну машину, де будемо вчитись і експериментувати. Пізніше в курсі буде і Staging, і автоматичний деплоймент на Production VPS.

**Ключові поняття:**

- **Environment (Середовище)** — ізольована інфраструктура для запуску певної версії застосунку.
- **Production (Prod)** — середовище з реальними користувачами і реальними даними. Будь-яка помилка тут — це downtime.
- **Staging** — точна копія Production для фінальної перевірки перед релізом.
- **Development (Dev)** — локальне середовище розробника, де дозволено все і ризики мінімальні.

### 2. Що таке віртуалізація?

До 2000-х на кожен сервіс — свій фізичний сервер. Потрібен web-сервер? Купи окремий комп'ютер. База даних? Ще один. Email? Ще. Все це займало величезні серверні кімнати, і більшість машин просто простоювала (типове завантаження — 5–15% потужності).

**Віртуалізація** вирішила цю проблему: на одному фізичному сервері можна запустити декілька **повністю ізольованих** операційних систем одночасно. Кожна думає, що вона «єдина» на машині, не знає про сусідів і не може на них вплинути.

```text
ФІЗИЧНЕ ЗАЛІЗО (CPU, RAM, Disk, Network)
         │
    ┌────▼────┐
    │Гіпервізор│  ← "Диригент" — розподіляє ресурси між VM
    └────┬────┘
    ┌────┼────┐
┌───▼──┐ ┌─▼───┐ ┌──▼──┐
│ VM 1 │ │ VM 2│ │ VM 3│   ← кожна — повноцінна ОС
│Ubuntu│ │ Win │ │CentOS│
└──────┘ └─────┘ └──────┘
```

**Ключові поняття:**

- **Віртуалізація** — технологія, що дозволяє запускати декілька ОС на одному фізичному гості.
- **Гіпервізор (Hypervisor)** — програмне забезпечення, що керує розподілом ресурсів між VM.
- **Virtual Machine (VM, Віртуальна машина)** — ізольована «комп'ютерна програма», що імітує повноцінний комп'ютер.

---

### 3. Гіпервізори: Тип 1 vs Тип 2

Існує два типи гіпервізорів — і вони принципово відрізняються архітектурою:

**Тип 1 — «Bare-Metal» (напряму на залізі)**

Встановлюється безпосередньо на фізичне залізо, **без хостової ОС**. Надзвичайно ефективний — немає зайвого прошарку. Використовується в корпоративних датацентрах і хмарних платформах.

```text
[ ЗАЛІЗО ]
    │
[ Гіпервізор Тип 1 ]  ← напряму, без Windows/Linux під ним
    │
[ VM 1 ] [ VM 2 ] ...
```

**Приклади:** VMware ESXi, Microsoft Hyper-V (server mode), KVM (Linux kernel).

**Тип 2 — «Hosted» (поверх хостової ОС)**

Встановлюється поверх звичайної ОС (Windows, macOS, Linux) як звичайна програма. Трохи менш ефективний через додатковий прошарок, але ідеальний для локальної розробки.

```text
[ ЗАЛІЗО ]
    │
[ Хостова ОС: Windows або macOS або Linux ]
    │
[ Гіпервізор Тип 2: VirtualBox / VMware Workstation ]
    │
[ VM: Ubuntu Server ]
```

**Приклади:** VirtualBox (безкоштовний, крос-платформний), VMware Workstation, Parallels (macOS).

| Характеристика     | Тип 1 (Bare-Metal)    | Тип 2 (Hosted)         |
| ------------------ | --------------------- | ---------------------- |
| Продуктивність     | ⭐⭐⭐ (максимальна)  | ⭐⭐ (є накладні)      |
| Де використовується| Датацентри, хмара     | Локальна розробка      |
| Потрібна хостова ОС| ❌ Ні                  | ✅ Так                  |
| Складність         | 🔴 Висока             | 🟢 Низька               |
| Приклади           | ESXi, KVM, Hyper-V   | VirtualBox, Parallels  |

**У нашому курсі:** ми використовуємо **VirtualBox (Тип 2)** — він безкоштовний, встановлюється на Windows/macOS/Linux та ідеально підходить для навчання.

**Ключові поняття:**

- **Bare-Metal Hypervisor (Тип 1)** — гіпервізор, що працює напряму на залізі без хостової ОС (ESXi, KVM).
- **Hosted Hypervisor (Тип 2)** — гіпервізор, що працює поверх хостової ОС як звичайна програма (VirtualBox).
- **VirtualBox** — безкоштовний гіпервізор Тип 2 від Oracle, крос-платформний.

### 4. Чим VM відрізняється від Docker-контейнера?

Це питання студенти обов'язково задають. Важливо пояснити різницю правильно, бо обидві технології «ізолюють» застосунки, але принципово різними способами.

**VM (Віртуальна машина):**

- Емулює **повний комп'ютер**: власне ядро ОС, власну пам'ять, власну мережеву карту.
- Повністю ізольована — навіть якщо хостова ОС впала, VM може продовжувати працювати.
- Займає більше ресурсів (сотні MB RAM на запуск).
- Запуск: хвилини.
- Підтримує `systemd`, власний мережевий стек, повноцінне init system.

**Docker-контейнер:**

- Ізолює лише **процеси** — використовує ядро хостової ОС (і тому працює тільки на Linux).
- Надлегкий — запускається за секунди, займає мегабайти.
- **Не** емулює повну ОС — немає `systemd`, немає власної мережевої карти.
- Ідеальний для мікросервісів та пакування застосунків.

```text
VM                           Docker
──────────────────────       ──────────────────────
│ App                │       │ App 1 │ App 2      │
│ Guest OS (Ubuntu)  │       │───────────────────│
│ Kernel             │       │ Docker Engine     │
│ Virtual Hardware   │       │ Host OS Kernel    │
│────────────────────│       │ Physical Hardware │
│ Hypervisor         │       ──────────────────────
│ Physical Hardware  │
──────────────────────
```

**Чому для цього курсу ми обираємо VM, а не Docker?**

Наш AI-агент (`openclaw`) запускається як `systemd`-сервіс. Він потребує:

- Повноцінного Linux kernel та init system (`systemd`)
- Власного мережевого стека (налаштування `iptables`, `firewall`)
- Ізоляції на рівні ОС (окремий системний користувач, `sudo`, власний `/home`)

Все це є у VM і **відсутнє** у Docker-контейнері. Тому для імітації «справжнього VPS» — лише VM.

> 💡 Docker ми також вивчимо — але пізніше (Тема 8), як інструмент для запуску **допоміжних сервісів** (наприклад, PostgreSQL), а не самого агента.

**Ключові поняття:**

- **Container (Контейнер)** — ізольований процес, що використовує ядро хостової ОС.
- **Docker** — найпопулярніша платформа контейнеризації.
- **systemd** — система ініціалізації Linux, відповідальна за запуск та керування сервісами; є у VM, але відсутня в Docker-контейнерах.

### 5. Infrastructure as Code: перший погляд

Ось проблема: студент налаштував VM вручну через GUI VirtualBox. Натискав кнопки, обирав параметри. Через тиждень його VM «зламалась». Що робити? Починати з нуля, знову натискаючи ті ж кнопки? А якщо потрібно передати VM колезі?

**Infrastructure as Code (IaC)** — це підхід, при якому конфігурація інфраструктури (у нашому випадку — VM) описується у текстовому файлі. Цей файл можна:

- зберегти в Git (і не втратити)
- передати колезі (він запустить **ідентично ту саму** VM)
- відновити за одну команду після краху

**Vagrant** — це інструмент, який перетворює `Vagrantfile` (текстовий опис VM) на реальну машину VirtualBox. Це і є IaC на локальному рівні.

```ruby
# Vagrantfile — декларативний опис нашої VM
Vagrant.configure("2") do |config|
  config.vm.box = "ubuntu/jammy64"   # Ubuntu 22.04 LTS
  config.vm.hostname = "devops-sandbox"

  config.vm.provider "virtualbox" do |vb|
    vb.memory = 2048  # 2 GB RAM
    vb.cpus = 2       # 2 ядра
  end
end
```

```bash
vagrant up    # Створює і запускає VM за описом з Vagrantfile
vagrant ssh   # Підключається до VM
vagrant halt  # Зупиняє VM
vagrant destroy  # Видаляє VM
```

> 🔑 **Ключовий інсайт:** Vagrantfile — це не «чарівний скрипт Vagrant». Це **перший практичний приклад IaC** в нашому курсі. Той самий принцип, який пізніше буде в Terraform (Тема 6) — тільки там для хмарних серверів, а тут — для локальних VM.

**Зв'язок з проєктом:** У репозиторії `devops-ai-assistant/10_Implementation/01_vm/` лежить готовий `Vagrantfile` для нашої Dev-машини. Студенти клонують репо і запускають `vagrant up` — і за хвилину отримують повноцінне Dev Environment, ідентичне у всіх членів команди.

**Ключові поняття:**

- **Infrastructure as Code (IaC)** — підхід, при якому інфраструктура описується як код у текстових файлах, доступних для версіонування та автоматизації.
- **Vagrant** — інструмент для автоматизованого створення та управління локальними VM на основі `Vagrantfile`.
- **Vagrantfile** — декларативний опис конфігурації VM у форматі Ruby DSL.
- **`vagrant up`** — команда, що читає `Vagrantfile` і матеріалізує описану VM у VirtualBox.
- **Box (Vagrant Box)** — готовий образ ОС для Vagrant (наприклад, `ubuntu/jammy64` = Ubuntu 22.04 LTS).

## ❓ Питання для обговорення на занятті

1. Уявіть, що ви — DevOps-інженер. Ваш колега-розробник хоче перевірити нову конфігурацію `nginx` прямо на Production, бо «так швидше». Як ви відреагуєте і чому?
2. Чим відрізняється ситуація «VM зламалась, налаштована вручну» від «VM зламалась, налаштована через Vagrantfile»? Яка з них безпечніша?
3. Ваш Tech Lead каже: «Нам не потрібен Staging — розробники і так тестують на своїх машинах». Що тут не так?
4. Де в нашому репозиторії ви вже бачили принцип «опис конфігурації в файлі» (IaC) — ще до того, як ми заговорили про Vagrant?
