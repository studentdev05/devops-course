## 🎯 Мета роботи

Налаштувати повний робочий процес з Git: створити форк репозиторію курсу, клонувати його на VM під правильним користувачем, налаштувати `origin` та `upstream`, створити особисту гілку, внести зміни та відправити їх у свій форк. У результаті студент матиме повний цикл роботи з Git, який використовується в реальних проєктах.

> **Контекст:** У Темі 4 ми налаштували безпарольний SSH-доступ під користувачем `openclaw`. Тепер ми використаємо це підключення для роботи з Git — саме так, як це робиться в реальних DevOps-проєктах.

## 🛠 Покрокова інструкція

### Крок 1: Підключення до VM під правильним користувачем

Перш ніж почати роботу з Git, потрібно переконатися, що ми працюємо під правильним користувачем — `openclaw`, а не `root`.

Виконайте на **хості** (вашому ноутбуці):

```bash
# Підключаємось до VM через SSH-псевдонім з Теми 4
ssh devvm-openclaw

# Перевіряємо, під яким користувачем ми
whoami
# Очікуваний результат: openclaw (НЕ root!)
```

**Якщо ви під `root`** — вийдіть і підключіться правильно:

```bash
exit
ssh devvm-openclaw
```

### Крок 2: Створення директорії для проєктів

Стандартна практика Linux — зберігати особисті проєкти в домашній директорії користувача.

```bash
# Створюємо директорію для проєктів
mkdir -p ~/projects

# Переходимо в неї
cd ~/projects

# Перевіряємо поточне розташування
pwd
# Очікуваний результат: /home/openclaw/projects
```

### Крок 3: Налаштування Git

Перш ніж працювати з комітами, Git повинен знати, хто ви є. Це налаштування виконується **один раз** на кожній новій машині.

```bash
# Вказуємо ім'я та email (ці дані з'являтимуться в кожному коміті)
git config --global user.name "Ваше Ім'я"
git config --global user.email "your@email.com"

# Перевіряємо результат
git config --global user.name
git config --global user.email
```

**Очікуваний результат:**

```text
Ваше Ім'я
your@email.com
```

**Що означає ця команда:**

- `git config --global user.name` — вказує Git ваше ім'я; воно відображатиметься як автор у кожному коміті
- `git config --global user.email` — вказує Git ваш email для підпису комітів
- `--global` — налаштування застосовуються для всього користувача `openclaw` на цій VM, а не лише для одного репозиторію

**Важливо:** якщо пропустити цей крок, `git commit` видасть помилку або створить коміт без автора.

### Крок 4: Створення форку репозиторію

Цей крок виконується в браузері на **хості** (не на VM).

1. Відкрийте сторінку репозиторію курсу на GitHub/GitLab (URL дасть викладач)
2. Натисніть кнопку **Fork** у верхньому правому куті
3. Виберіть свій акаунт як місце для форку
4. Дочекайтеся завершення — тепер у вас є власна копія репозиторію

**Що тут відбувається:**

- **Fork** — це не клон на ваш комп'ютер, а окрема копія репозиторію у вашому акаунті на GitHub/GitLab
- Після форку ви отримуєте власний віддалений репозиторій, куди ви маєте право виконувати `push`
- Оригінальний репозиторій викладача залишається недоторканим

### Крок 5: Клонування форку на VM

Повертаємось до терміналу на VM.

```bash
# Переконайтеся, що ви в ~/projects
cd ~/projects

# Клонуємо ВАШ форк (замініть YOUR_USERNAME на ваше ім'я користувача GitHub)
git clone https://github.com/YOUR_USERNAME/devops-course.git

# Переходимо в директорію проєкту
cd devops-course

# Перевіряємо, де ми знаходимося
pwd
# Очікуваний результат: /home/openclaw/projects/devops-course
```

**Що означає ця команда:**

- `git clone` — створює локальну копію віддаленого репозиторію на диску VM
- `YOUR_USERNAME` у URL — це ваш акаунт, тобто ми клонуємо **свій форк**, а не репозиторій викладача

### Крок 6: Перевірка remote-налаштувань

Після клонування Git автоматично створює remote з ім'ям `origin`.

```bash
# Перевіряємо налаштування remote
git remote -v
```

**Очікуваний результат:**

```text
origin https://github.com/YOUR_USERNAME/devops-course.git (fetch)
origin https://github.com/YOUR_USERNAME/devops-course.git (push)
```

**Що це означає:**

- `origin` — це ваш форк, куди ви маєте право писати
- Поки що ми бачимо тільки `origin` — `upstream` треба додати окремо

---

### Крок 7: Додавання upstream

Щоб отримувати оновлення від викладача, потрібно додати оригінальний репозиторій як `upstream`.

```bash
# Додаємо оригінальний репозиторій як upstream
# Замініть INSTRUCTOR на ім'я користувача викладача
git remote add upstream https://github.com/INSTRUCTOR/devops-course.git

# Перевіряємо результат
git remote -v
```

**Очікуваний результат:**

```text
origin https://github.com/YOUR_USERNAME/devops-course.git (fetch)
origin https://github.com/YOUR_USERNAME/devops-course.git (push)
upstream https://github.com/INSTRUCTOR/devops-course.git (fetch)
upstream https://github.com/INSTRUCTOR/devops-course.git (push)
```

**Що це означає:**

- `origin` — ваш форк (куди ви відправляєте свої зміни)
- `upstream` — репозиторій викладача (звідки ви отримуєте оновлення)

### Крок 8: Створення особистої гілки

Ніколи не працюйте напряму в `main` — створіть окрему гілку для своїх змін.

```bash
# Створюємо нову гілку і перемикаємось на неї
git checkout -b my-topic5-work

# Перевіряємо поточну гілку
git branch
# Очікуваний результат: * my-topic5-work (зірочка позначає поточну гілку)
```

**Що означає ця команда:**

- `git checkout -b` — одночасно створює нову гілку і перемикає на неї
- `my-topic5-work` — це ім'я гілки; ви можете вибрати інше

---

### Крок 9: Внесення зміни

Тепер зробимо маленьку зміну, щоб побачити повний цикл роботи з Git.

```bash
# Додаємо свій рядок у README.md
echo "Студент: Ваше Ім'я" >> README.md

# Перевіряємо статус
git status
```

**Очікуваний результат:**

```text
On branch my-topic5-work
Changes not staged for commit:
  modified: README.md
```

**Що показує `git status`:**

- У якій гілці ви знаходитесь
- Які файли змінені
- Які файли ще не підготовлені до коміту

---

### Крок 10: Додавання файлу до staging area

Перш ніж закомітити зміни, потрібно додати їх до staging area.

```bash
# Додаємо README.md до staging area
git add README.md

# Перевіряємо статус
git status
```

**Очікуваний результат:**

```text
On branch my-topic5-work
Changes to be committed:
  modified: README.md
```

**Що змінилося:**

- Файл тепер у секції "Changes to be committed" — готовий до коміту
- Якщо ви передумали — можна виконати `git reset README.md`

---

### Крок 11: Створення коміту

Тепер фіксуємо зміни в історії Git.

```bash
# Створюємо коміт з повідомленням
git commit -m "Додано інформацію про студента"

# Перевіряємо історію
git log --oneline -3
```

**Очікуваний результат:**

```text
abc1234 (HEAD -> my-topic5-work) Додано інформацію про студента
def5678 (origin/main, origin/HEAD, main) Початковий коміт
```

**Що означає ця команда:**

- `git commit` — створює зафіксований знімок змін
- `-m "..."` — повідомлення коміту (короткий опис змін)

---

### Крок 12: Захист секретів — файл `.gitignore`

Уявіть, що ви випадково створили файл `.env` з конфіденційними даними. Без `.gitignore` Git побачить цей файл і запропонує його закомітити — що є серйозним порушенням безпеки.

```bash
# Симулюємо секретний файл
echo "API_KEY=super_secret_12345" > .env

# Перевіряємо — Git бачить цей файл
git status
```

**Очікуваний результат:**

```text
Untracked files:
  .env
```

Git бачить `.env`. Вирішимо це за допомогою `.gitignore`:

```bash
# Створюємо .gitignore і додаємо .env до списку ігнорованих файлів
echo ".env" > .gitignore

# Перевіряємо статус
git status
```

**Очікуваний результат:**

```text
Untracked files:
  .gitignore
```

Тепер `.env` зник зі статусу — Git його ігнорує. Натомість з'явився сам `.gitignore`, який потрібно закомітити:

```bash
# Додаємо і комітимо .gitignore
git add .gitignore
git commit -m "Додано .gitignore для захисту секретів"
```

**Що ми зробили:**

- Створили `.gitignore` — файл зі списком шаблонів файлів, які Git не відстежуватиме
- Закомітили `.gitignore` — тепер він теж під контролем версій і захищатиме репозиторій у майбутньому

### Крок 13: Відправка змін у ваш форк

Тепер відправляємо локальну гілку у ваш віддалений репозиторій.

```bash
# Відправляємо гілку my-topic5-work у origin
git push origin my-topic5-work
```

**Очікуваний результат:**

```text
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Writing objects: 100% (3/3), 287 bytes | 287.00 KiB/s, done.
To https://github.com/YOUR_USERNAME/devops-course.git
 * [new branch]      my-topic5-work -> my-topic5-work
```

**Що означає ця команда:**

- `git push` — відправляє локальні зміни у віддалений репозиторій
- `origin` — це ваш форк (куди ви маєте право писати)
- `my-topic5-work` — ім'я гілки, яку ви відправляєте

### Крок 15: Додатково — як підтягнути майбутні оновлення від викладача

Цей крок потрібен не для першого `clone`, а для реальної роботи протягом семестру.

Уявіть ситуацію: через тиждень викладач оновив основний репозиторій курсу — додав нові файли, виправив помилки або змінив інструкції. У вашому форку та у вашій локальній копії цих змін ще немає.

Саме для цього ми раніше додали `upstream`: він вказує на оригінальний репозиторій викладача, звідки можна отримувати оновлення.

Перш ніж забирати оновлення від викладача, спочатку перейдіть у свою локальну основну гілку `main`. Це важливо, тому що зміни викладача з'являються саме в `main`, і нам потрібно спочатку оновити свою базову гілку, а вже потім за потреби створювати від неї нові робочі гілки.

Логіка така:

- `git checkout main` — перемикає вас у локальну основну гілку;
- `git fetch upstream` — забирає нові коміти з репозиторію викладача, але ще не вбудовує їх у вашу поточну гілку;
- `git merge upstream/main` — вбудовує отримані зміни у вашу локальну гілку `main`;
- `git push origin ...` — відправляє вже оновлений результат у ваш форк.

Тобто `fetch` — це «забрати інформацію», а `merge` — це «включити ці зміни у свою роботу».

Виконайте:

```bash
# Переходимо в локальну основну гілку
git checkout main

# Отримуємо зміни з upstream (але ще не застосовуємо їх)
git fetch upstream

# Переглядаємо, які коміти є в репозиторії викладача
git log upstream/main --oneline -5

# Вбудовуємо ці оновлення у свою локальну гілку
git merge upstream/main
```

**Що має відбутися:**

- після `git fetch upstream` ваш локальний репозиторій "дізнається" про нові коміти викладача;
- після `git merge upstream/main` ці зміни з'являються у вашій локальній гілці `main`;
- якщо merge пройшов успішно, ви можете відправити оновлений результат у свій форк.

Наприклад, якщо ви хочете синхронізувати основну гілку:

```bash
git checkout main
git fetch upstream
git merge upstream/main
git push origin main
```

> 💡 **Навіщо це вам:** якщо викладач оновить основний репозиторій курсу, цих змін не буде автоматично ні у вашому форку, ні у вашій локальній копії. Саме через `upstream` ви зможете забрати ці оновлення, вбудувати їх у свою локальну гілку і потім відправити оновлений результат у свій форк.

## ❓ Контрольні питання

1. Ви виконали `git clone` репозиторію викладача (не свій форк). При `git push` отримали помилку `Permission denied`. Поясніть причину та як це виправити.
2. Студент каже: «Я зробив fork, значить проєкт уже є на моїй VM». Чи це правда? Поясніть різницю між fork та clone.
3. Навіщо потрібен `upstream`? Чи можна працювати без нього? Коли `upstream` стає необхідним?
4. Ви випадково закомітили `.env` з API-ключем у публічний репозиторій. Потім видалили файл і зробили новий коміт. Чи достатньо цього? Поясніть.
5. Студент працює під `root` і каже: «Так простіше». Поясніть, чому це погана практика з точки зору DevOps.
6. Ви хочете відмінити `git add README.md` до коміту. Яка команда для цього потрібна? А як відкотити вже зроблений коміт?
